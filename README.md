# Игра "Лабиринт" (вариант 2.1, многопроцессная реализация)

Сетевая клиент-серверная игра "Лабиринт" с компьютерным противником.
Один и тот же бинарник в зависимости от флага запуска работает либо в
режиме TCP-сервера (`-s`), либо в режиме TCP-клиента (`-c`) — без
перекомпиляции.

Реализовано на C++17 поверх Berkeley-сокетов (`<sys/socket.h>`).
Сервер многопроцессный, на каждое подключение делается `fork()`,
дочерний процесс ведёт игру с одним клиентом и завершается. Родительский
процесс продолжает принимать новые подключения; zombie-процессы собираются
обработчиком `SIGCHLD`. Все сокеты — в блокирующем режиме.

## Сборка и запуск

```bash
make            # сборка → ./maze
make clean      # удалить артефакты
make docs       # сгенерировать Doxygen-документацию
make diagrams   # сгенерировать диаграммы
```

Запуск сервера:

```bash
./maze -s [-p порт] [-m лимит_ходов] [-z размер] [-w стен]
```

Запуск клиента:

```bash
./maze -c -n <имя> [-a <ip-сервера>] [-p порт] [-t]
# -t — режим тестирования производительности (без лимита ходов)
```

Управление клиентом: `w` — вперёд, `s` — назад, `a` — влево,
`d` — вправо, `q` — сдаться.

## Протокол прикладного уровня

- Транспорт: **TCP**, порт по умолчанию 4321.
- Кодировка: ASCII, текстовый.
- Каждое сообщение завершается символом `\n`.
- Разбор — потокенно по пробелам; лишние пробелы игнорируются.
- После `WIN` / `LOSE` / `ERR` сервер закрывает соединение.

### Команды клиента

| Сообщение         | Назначение                                        |
|-------------------|---------------------------------------------------|
| `HELLO <имя>`     | начало игры в обычном режиме                      |
| `PERF <имя>`      | начало игры в режиме тестирования (без лимита)    |
| `FORWARD`         | ход вперёд (+Y)                                   |
| `BACK`            | ход назад (-Y)                                    |
| `LEFT`            | ход влево (-X)                                    |
| `RIGHT`           | ход вправо (+X)                                   |
| `GIVEUP`          | сдаться                                           |

### Ответы сервера

| Сообщение                  | Назначение                                            |
|----------------------------|-------------------------------------------------------|
| `SIZE <n>`                 | размер поля (отправляется один раз после HELLO/PERF)  |
| `OK <x> <y> <ходов>`       | успешный ход, новая позиция и остаток ходов           |
| `WALL <ходов>`             | на пути стена, остаток ходов                          |
| `WIN`                      | игрок дошёл до цели                                   |
| `LOSE`                     | игрок проиграл (исчерпал ходы или сдался)             |
| `ERR <текст>`              | ошибка протокола / некорректное сообщение клиента     |

Значение `<ходов>` в режиме тестирования (`PERF`) равно `-1`
(безлимитный режим).

## Архитектура и обоснование структур данных

### Модель `Maze`

- Поле — квадрат `size × size`. Игрок стартует в `(0, 0)`, цель —
  `(size-1, size-1)`.
- Множество стен — `std::set<Wall>`:
  - проверка наличия стены — `O(log N)`;
  - дубликаты исключены автоматически;
  - стена нормализуется (`from < to`), чтобы порядок указания клеток не
    влиял на ключ.
- `std::mt19937` — современный, быстрый, детерминированный (при заданном
  seed) генератор для случайного выбора стен.
- Генерация стен — алгоритм с откатом: стены добавляются по одной из
  случайной перестановки кандидатов; после каждой стены проверяется
  достижимость цели обходом в глубину (`Utils::dfsPathExists`). Если
  путь пропал — стена снимается. Это гарантирует, что игра всегда
  проходима за конечное число ходов.

### Сервер `Server`

- Слушающий TCP-сокет, `accept()` в бесконечном цикле.
- На каждое подключение — `fork()`. Дочерний процесс закрывает слушающий
  сокет и запускает `Session::run()`, родитель — закрывает клиентский
  сокет и продолжает `accept`.
- `SIGCHLD` обрабатывается с флагом `SA_NOCLDSTOP`: в обработчике
  собираются все завершившиеся потомки через `waitpid(-1, …, WNOHANG)`,
  что предотвращает накопление zombie-процессов.
- `SIGPIPE` игнорируется, чтобы разрыв соединения с клиентом не убивал
  процесс (ошибка возвращается через `send/recv`).
- `SO_REUSEADDR` — для быстрого перезапуска без `TIME_WAIT`-задержки.
- Все сокеты — в блокирующем режиме (требование варианта 2.1).

### Сессия `Session`

- Создаётся в дочернем процессе после `fork()`.
- Читает приветственное `HELLO/PERF`, генерирует лабиринт, отправляет
  `SIZE`, далее цикл «принять команду → обновить модель → отправить
  ответ» до `WIN/LOSE/ERR`.
- Буферизованный построчный приём: пока в буфере нет `\n`,
  накапливается полученное; затем извлекается одна полная строка.

### Клиент `Client`

- TCP-соединение, отправка `HELLO/PERF`, приём `SIZE`, настройка
  локальной модели `Maze` того же размера для визуализации.
- Терминал переводится в неканонический режим без эха (одна клавиша —
  один ход) и восстанавливается в деструкторе.
- При ответе `WALL` клиент добавляет стену в локальную модель, чтобы
  в дальнейших ходах она была видна на карте.
- История последних 6 ходов выводится под полем.

### Протокол (`Protocol.hpp`)

Константы команд/ответов, помощники для формирования сообщений
(`makeHello`, `makeOk`, `makeWall`, …) и `tokenize()` для разбора строки
на токены вынесены в одно место. Это исключает «магические» строки в
коде сервера и клиента и упрощает изменение формата.

## Тестирование производительности

Команда `PERF <имя>` переводит сессию в режим без лимита ходов
(`moves_left = -1`). Ходы не уменьшают счётчик, поражение наступает
только по команде `GIVEUP`. Это позволяет нагружать сервер длинными
последовательностями команд для измерения throughput. Клиент включает
этот режим флагом `-t`.

## Проверка по локальной сети

1. На сервере: узнать IP — `ip addr` (Linux) / `ifconfig` (macOS).
2. Запустить сервер: `./maze -s -p 4321`.
3. На клиенте: проверить связь — `ping <ip-сервера>`.
4. Запустить клиента: `./maze -c -a <ip-сервера> -p 4321 -n <имя>`.

